---
title: "Cancel an Open Order — DELETE /orders/{order_id}"
sidebarTitle: "Cancel Order"
description: "Request cancellation of an open order by its ID. Returns cancel_requested status; the order may still fill before the cancel is processed."
---

This endpoint submits a cancellation request for an open order identified by its order ID. Cancellation is asynchronous — the server accepts and queues the cancel request, but the order may still be filled before the cancellation is processed. A successful response confirms only that the cancel was accepted, not that the order has been canceled. This endpoint requires the `trade:cancel` scope. You may optionally supply an `Idempotency-Key` header to make the request safely retriable.

## Request

**`DELETE /orders/{order_id}`**

**Base URL:** `https://test-eu.polycopy.fun/openapi/v1`

**Required scope:** `trade:cancel`

### Authentication Headers

All requests must include the following HMAC-signed headers:

| Header | Type | Description |
|---|---|---|
| `X-API-Key` | string | Your API key identifier |
| `X-Timestamp` | string | Unix timestamp of the request |
| `X-Nonce` | string | Unique nonce string for this request |
| `X-Signature` | string | HMAC signature computed over the request |

### Optional Header

<ParamField header="Idempotency-Key" type="string">
  A unique key (max 128 characters) that makes this request safely retriable. If the server has already accepted a cancel request with this key, it returns the original response without submitting a duplicate cancel.
</ParamField>

### Path Parameters

<ParamField path="order_id" type="string" required>
  The unique order ID of the order you want to cancel (e.g., `0xorder_id_here`).
</ParamField>

## Response

All responses use the standard envelope. On success, the `data` object contains the following fields:

<ResponseField name="order_id" type="string">
  The ID of the order for which a cancellation was requested.
</ResponseField>

<ResponseField name="status" type="string">
  Always `cancel_requested` on a successful response. This indicates the cancel request was accepted by the server, not that the order has been canceled yet.
</ResponseField>

<Note>
  A `status` of `cancel_requested` means the cancellation was **accepted**, not that the order is already canceled. Because cancellation is asynchronous, the order may still fill before the cancel is processed. Poll `GET /orders/{order_id}` to confirm the final status of the order.
</Note>

## Error Codes

| Code | Description |
|---|---|
| `ORDER_CANCEL_FAILED` | The cancellation request could not be submitted due to an internal or exchange-side error. |
| `ORDER_NOT_FOUND` | No order was found with the given ID, or the order belongs to a different account. |

## Example

### Cancel an Open Order

```bash
curl -X DELETE "https://test-eu.polycopy.fun/openapi/v1/orders/0xorder_id_here" \
  -H "X-API-Key: pk_your_key_id" \
  -H "X-Timestamp: 1717200000" \
  -H "X-Nonce: abc123" \
  -H "X-Signature: <signature>"
```

**Response:**

```json
{
  "code": "OK",
  "message": "success",
  "data": {
    "order_id": "0xorder_id_here",
    "status": "cancel_requested"
  },
  "request_id": "req_202606011200000001"
}
```

### Confirm Final Cancellation

After receiving `cancel_requested`, poll the order until `status` transitions away from `open`:

```bash
curl -X GET "https://test-eu.polycopy.fun/openapi/v1/orders/0xorder_id_here" \
  -H "X-API-Key: pk_your_key_id" \
  -H "X-Timestamp: 1717200000" \
  -H "X-Nonce: abc123" \
  -H "X-Signature: <signature>"
```

<Warning>
  Do not assume an order is canceled just because you received `cancel_requested`. Always verify the final status via `GET /orders/{order_id}` before taking downstream actions that depend on the order being closed.
</Warning>
