---
title: "GET /afk/records — List AFK Strategy Execution Records"
sidebarTitle: "AFK Records"
description: "Retrieve AFK strategy execution records with cursor pagination. Filter by strategy_id or status to audit trade history and diagnose failures."
---

AFK records capture every trade attempt executed by your AFK strategies. Use this endpoint to audit execution history, monitor performance, and diagnose failures. Each record corresponds to a single order attempt by a strategy, and includes the price, amount, on-chain transaction hash, and final fill status. When a record has `status=2` (failed), the `error_msg` field contains the failure reason so you can quickly identify and address the issue. Requires the `afk:read` scope.

## Request

```http
GET /afk/records
```

**Base URL:** `https://test-eu.polycopy.fun/openapi/v1`

**Required scope:** `afk:read`

<Note>
  All requests must include the four HMAC-signed authentication headers: `X-API-Key`, `X-Timestamp`, `X-Nonce`, and `X-Signature`.
</Note>

## Query Parameters

<ParamField query="strategy_id" type="integer">
  Filter records to a single AFK strategy by its numeric ID.
</ParamField>

<ParamField query="status" type="integer">
  Filter records by execution status. See the status reference table below.
</ParamField>

<ParamField query="limit" type="integer" default="50">
  Number of records to return per page. Accepted range: `1`–`200`.
</ParamField>

<ParamField query="cursor" type="string">
  Pagination cursor. Pass the `next_cursor` value from a previous response to fetch the next page of results.
</ParamField>

## Record Status Reference

| Status | Label | Description |
|--------|-------|-------------|
| `1` | Success | The order was filled successfully. |
| `2` | Failed | The order failed. Check `error_msg` for the reason. |
| `3` | Waiting | The order has been submitted and is awaiting execution. |
| `4` | Canceled | The order was canceled before it could be filled. |
| `5` | Partial Fill | The order was partially filled. |

## Pagination

This endpoint uses cursor-based pagination. When a response contains a non-null `next_cursor` value, pass it as the `cursor` query parameter to retrieve the next page. Continue until `next_cursor` is `null`, which indicates you have retrieved all matching records.

```
GET /afk/records?strategy_id=42&limit=20               → page 1, returns next_cursor="cur_abc"
GET /afk/records?strategy_id=42&limit=20&cursor=cur_abc → page 2, returns next_cursor=null
```

## Example Request

```bash
curl -X GET "https://test-eu.polycopy.fun/openapi/v1/afk/records?strategy_id=42&status=1&limit=20" \
  -H "X-API-Key: pk_your_key_id" \
  -H "X-Timestamp: 1717200000" \
  -H "X-Nonce: abc123" \
  -H "X-Signature: <signature>"
```
