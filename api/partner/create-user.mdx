---
title: "POST /partner/users — Create or Get a Partner User"
sidebarTitle: "Create User"
description: "Provision a new partner user or retrieve an existing one by external_user_id. Returns the user account and wallet initialization status."
---

This endpoint is an idempotent create-or-get operation: if you submit an `external_user_id` that already exists under your partner account, the existing user record is returned rather than creating a duplicate. This makes it safe to call multiple times — for example, when polling for wallet initialization. You must authenticate with a partner API key that carries the `partner:users:manage` or `*` scope.

## Request

```
POST https://test-eu.polycopy.fun/openapi/v1/partner/users
```

### Required Scope

`partner:users:manage` or `*`

### Request Body

<ParamField body="external_user_id" type="string" required>
  Your stable internal identifier for this user. This value is used as the idempotency key — submitting the same `external_user_id` twice will return the existing user rather than creating a new one.
</ParamField>

<ParamField body="display_name" type="string">
  A human-readable name for the user. Used for display purposes only.
</ParamField>

<ParamField body="email" type="string">
  The user's email address.
</ParamField>

## Wallet Status Lifecycle

After a user is provisioned, their wallet is initialized asynchronously. The `wallet_status` field in the response reflects the current state of that process:

| Status | Meaning |
|--------|---------|
| `pending` | The wallet is still being initialized. Call this endpoint again to check for an update. |
| `ready` | The wallet has been successfully initialized. You can now create API keys for this user. |
| `failed` | Wallet initialization failed. Call this endpoint again to trigger a re-attempt. |

<Note>
  You must confirm `wallet_status` is `ready` before creating a user API key. Attempting to create an API key for a user with a `pending` or `failed` wallet will not succeed.
</Note>

## Response

All responses use the standard envelope: `code`, `message`, `data`, and `request_id`. The `data` object contains the user record described below.

<ResponseField name="partner_account_id" type="string">
  The account ID of the partner that provisioned this user.
</ResponseField>

<ResponseField name="user_account_id" type="string">
  The unique account ID assigned to this user by the platform.
</ResponseField>

<ResponseField name="external_user_id" type="string">
  Your stable internal identifier for this user, echoed back from the request.
</ResponseField>

<ResponseField name="display_name" type="string">
  Human-readable name for the user.
</ResponseField>

<ResponseField name="email" type="string">
  Email address associated with the user account.
</ResponseField>

<ResponseField name="status" type="string">
  Account status. One of `active` or `disabled`.
</ResponseField>

<ResponseField name="wallet_status" type="string">
  Wallet initialization status. One of `pending`, `ready`, or `failed`. See the lifecycle table above.
</ResponseField>

<ResponseField name="wallet_created" type="boolean">
  Indicates whether a wallet has been created for this user.
</ResponseField>

<ResponseField name="created_at" type="string">
  ISO 8601 timestamp of when the user record was created.
</ResponseField>

<ResponseField name="updated_at" type="string">
  ISO 8601 timestamp of the most recent update to the user record.
</ResponseField>

## Example Request

```bash
curl -X POST "https://test-eu.polycopy.fun/openapi/v1/partner/users" \
  -H "X-API-Key: pk_your_key_id" \
  -H "X-Timestamp: 1717200000" \
  -H "X-Nonce: abc123" \
  -H "X-Signature: <signature>" \
  -H "Content-Type: application/json" \
  -d '{"external_user_id": "user_123", "display_name": "Alice", "email": "alice@example.com"}'
```
