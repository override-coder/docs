Use this endpoint to generate a user-level API key for a provisioned partner user. The key allows that user to authenticate against the Poly Trader API independently using their own credentials and scopes. The `secret` value in the response is shown **only once** — it cannot be retrieved again after this call. You must authenticate with a partner API key that carries the `partner:api_keys:manage` or `*` scope.

## Request

```
POST https://test-eu.polycopy.fun/openapi/v1/partner/users/{user_account_id}/api-keys
```

### Required Scope

`partner:api_keys:manage` or `*`

### Path Parameters

<ParamField path="user_account_id" type="string" required>
  The unique account ID of the user for whom you are creating the key. Must match the pattern `^99[0-9]{20}$`. This value is returned in the `user_account_id` field when you create or retrieve a user.
</ParamField>

### Request Body

<ParamField body="name" type="string" required>
  A human-readable label for this API key. Use something descriptive so you can identify the key later (e.g. `"user trading key"`).
</ParamField>

<ParamField body="scopes" type="array of strings">
  The list of permission scopes to grant this key. If omitted, the key is created with the default set of scopes: `trade:read`, `trade:order`, `trade:cancel`, `wallet:read`, `positions:read`, `copytrade:read`, `copytrade:manage`, `afk:read`, `afk:manage`.

  The `*` wildcard and the `withdraw:create` scope are **not** permitted for user API keys in v1.
</ParamField>

## Available User Key Scopes

| Scope | Description |
|-------|-------------|
| `trade:read` | Read trading history, open orders, and market data |
| `trade:order` | Place new orders |
| `trade:cancel` | Cancel open orders |
| `wallet:read` | View wallet balances and transaction history |
| `positions:read` | Read open and historical positions |
| `copytrade:read` | View copy-trading configurations and stats |
| `copytrade:manage` | Create and manage copy-trading settings |
| `afk:read` | View AFK (automated) strategy configurations |
| `afk:manage` | Create and manage AFK strategies |

<Warning>
  The `secret` is shown only once at creation time. Store it immediately in a secure location — it cannot be recovered after this response. If lost, you must create a new API key.
</Warning>

<Note>
  The user must have `wallet_status=ready` before you can create an API key for them. If the user's wallet is still `pending` or has `failed`, call `POST /partner/users` again to check for an updated status or trigger a re-attempt, then retry this request.
</Note>

## Response

All responses use the standard envelope: `code`, `message`, `data`, and `request_id`. The `data` object contains the fields described below.

<ResponseField name="key_id" type="string">
  The unique identifier for the newly created API key (e.g. `pk_abc123`).
</ResponseField>

<ResponseField name="name" type="string">
  The label you provided for this key.
</ResponseField>

<ResponseField name="status" type="string">
  The current status of the key. Newly created keys are `active`.
</ResponseField>

<ResponseField name="scopes" type="array of strings">
  The permission scopes assigned to this key.
</ResponseField>

<ResponseField name="ip_whitelist" type="array">
  List of IP addresses or CIDR ranges permitted to use this key. Empty array means no IP restriction is applied.
</ResponseField>

<ResponseField name="secret" type="string">
  The key secret. **Shown only once.** Store this value securely before discarding the response.
</ResponseField>

<ResponseField name="last_used_at" type="string | null">
  ISO 8601 timestamp of the last time this key was used to authenticate a request. `null` for a brand-new key.
</ResponseField>

<ResponseField name="created_at" type="string">
  ISO 8601 timestamp of when the API key was created.
</ResponseField>

## Example Request

```bash
curl -X POST "https://test-eu.polycopy.fun/openapi/v1/partner/users/9987654321098765432100/api-keys" \
  -H "X-API-Key: pk_your_partner_key" \
  -H "X-Timestamp: 1717200000" \
  -H "X-Nonce: abc123" \
  -H "X-Signature: <signature>" \
  -H "Content-Type: application/json" \
  -d '{"name": "user trading key", "scopes": ["trade:order", "wallet:read"]}'
```
