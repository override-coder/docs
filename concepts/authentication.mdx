PolyCop OpenAPI does not use sessions or Bearer tokens. Every request must include an API key ID and an HMAC-SHA256 signature.

## Required headers

Send these four headers on every request:

<ParamField header="X-API-Key" type="string" required>
  Your API key ID, in the format `pk_...`.
</ParamField>

<ParamField header="X-Timestamp" type="integer" required>
  Unix timestamp in seconds. It must be within +/-30 seconds of the server time.
</ParamField>

<ParamField header="X-Nonce" type="string" required>
  A random string that must be unique within the timestamp window. Use at least
  16 random bytes, hex encoded.
</ParamField>

<ParamField header="X-Signature" type="string" required>
  The HMAC-SHA256 hex digest of the canonical string.
</ParamField>

## Canonical string

Before you compute `X-Signature`, build the canonical string. The server builds the same string and compares signatures.

The canonical string format is:

```
METHOD + "\n" + PATH + "\n" + QUERY + "\n" + TIMESTAMP + "\n" + NONCE + "\n" + BODY_SHA256
```

Each component is defined as follows:

| Component | Description |
|---|---|
| `METHOD` | Uppercase HTTP verb: `GET`, `POST`, `PATCH`, or `DELETE`. |
| `PATH` | The full request path including the `/openapi/v1` prefix, e.g. `/openapi/v1/wallet`. |
| `QUERY` | The URL query string, without the leading `?`. Use an empty string when there is no query string. The blank line is still part of the signature. |
| `TIMESTAMP` | The exact same integer value you send in the `X-Timestamp` header. |
| `NONCE` | The exact same string value you send in the `X-Nonce` header. |
| `BODY_SHA256` | The SHA-256 hex digest of the **raw request body bytes**. For requests with no body (e.g. `GET`), compute the SHA-256 of the empty string: `e3b0c44298fc1c149afbf4c8996fb92427ae41e4649b934ca495991b7852b855`. |

Once you have built the canonical string, sign it with your API secret key:

```
X-Signature = HMAC-SHA256(secret_key, canonical_string).hex()
```

<Note>
  Hash the raw request body bytes. Do not hash a pretty-printed or re-serialized
  version of the body.
</Note>

## Canonical string examples

### GET request with no body or query string

For `GET /openapi/v1/wallet` with timestamp `1717200000` and nonce `abc123randomnonce`:

```
GET
/openapi/v1/wallet

1717200000
abc123randomnonce
e3b0c44298fc1c149afbf4c8996fb92427ae41e4649b934ca495991b7852b855
```

The third line is blank because there is no query string. `BODY_SHA256` is the SHA-256 hash of an empty string.

### POST request with a JSON body

For `POST /openapi/v1/orders` with body `{"asset":"abc","side":"buy","amount_usd":"10"}`, timestamp `1717200001`, and nonce `xyz789anothernonce`:

```
POST
/openapi/v1/orders

1717200001
xyz789anothernonce
<sha256 of body bytes>
```

Replace `<sha256 of body bytes>` with the SHA-256 hex digest of the UTF-8 encoded body before signing.

### GET request, with query string

For `GET /openapi/v1/orders?locale=en`, the query component is `locale=en`:

```
GET
/openapi/v1/orders
locale=en
1717200002
def456yetanothernonce
e3b0c44298fc1c149afbf4c8996fb92427ae41e4649b934ca495991b7852b855
```

## Timestamp skew

The server enforces a strict clock-skew window to prevent replay attacks:

- Your `X-Timestamp` must be within +/-30 seconds of the server's current Unix time.
- If the timestamp falls outside this window, the server rejects the request with `API_KEY_TIMESTAMP_EXPIRED`.
- Use a time-synchronised clock source (e.g., NTP) on your signing host to avoid drift. Cloud environments typically synchronise automatically; embedded or local systems may require explicit NTP configuration.

## Nonce rules

A nonce ("number used once") prevents an attacker who intercepts a valid request from replaying it:

- Your nonce must be **unique within the same timestamp window**. Two requests with the same `X-Timestamp` value and the same `X-Nonce` value will conflict.
- Replaying a nonce within the timestamp window is rejected with `API_KEY_NONCE_REPLAYED`.
- Generate nonces using a **cryptographically secure random source**, for example, 16 or more random bytes encoded as a lowercase hex string (resulting in a 32+ character string).

<Tip>
  A UUID v4 works well as a nonce. It is random enough for this use and easy to
  find in logs.
</Tip>

## Retry behaviour

<Warning>
  **Every retry must use a fresh `X-Timestamp`, `X-Nonce`, and `X-Signature`.**
  You cannot reuse a signature from a previous attempt, even for idempotent
  requests. The old timestamp may have expired and the old nonce is considered
  consumed. When retrying a mutation you want to be idempotent, reuse the same
  `Idempotency-Key` header while generating brand-new values for all four
  signature headers.
</Warning>

See [Idempotency](/concepts/idempotency) for guidance on safely retrying state-changing requests.

## API key scopes

Each API key has scopes that control which endpoints it can call. If the key does not have the required scope, the API returns `API_KEY_LIMIT_EXCEEDED`.

| Scope | Description |
|---|---|
| `trade:read` | Read orders |
| `trade:order` | Place orders |
| `trade:cancel` | Cancel orders |
| `wallet:read` | Read wallet balances |
| `withdraw:create` | Create Polygon withdrawals |
| `positions:read` | Read positions |
| `copytrade:read` | Read copytrade configs and records |
| `copytrade:manage` | Create, update, and delete copytrade configs |
| `afk:read` | Read AFK strategies and records |
| `afk:manage` | Create and update AFK strategies |

<Note>
  Withdrawals also require a valid 2FA code in the request body. Use your
  account security flow to set up 2FA before creating withdrawals.
</Note>

## Authentication error codes

If authentication fails, the server returns one of the following error codes:

| Error code | Meaning |
|---|---|
| `UNAUTHORIZED` | The request lacks valid authentication credentials. |
| `API_KEY_LIMIT_EXCEEDED` | The key does not have the scope required for this endpoint. |
| `API_KEY_INVALID` | The `X-API-Key` value does not correspond to any known API key. |
| `API_KEY_DISABLED` | The API key exists but has been disabled. |
| `API_KEY_SIGNATURE_INVALID` | The `X-Signature` does not match the server's recomputed HMAC-SHA256 of the canonical string. |
| `API_KEY_TIMESTAMP_EXPIRED` | The `X-Timestamp` is more than 30 seconds in the past or future. |
| `API_KEY_NONCE_REPLAYED` | The `X-Nonce` has already been used within the current timestamp window. |
